## 2. Определения

### BPM

Business Process Management — направление управления бизнес-процессами: описание, анализ, оптимизация, автоматизация и контроль процессов.

### BPMN

Business Process Model and Notation — нотация моделирования бизнес-процессов, то есть стандартный язык описания процессов.

### ARIS

ARIS — профессиональная среда и методологическая система для моделирования процессов, функций, данных, организационной структуры и архитектуры предприятия.

### Что входит в категорию

- описание и моделирование процессов;
- анализ и оптимизация процессов;
- процессная архитектура;
- связь процессов с ролями, данными и документами;
- подготовка процессов к автоматизации.

### Что не входит

- простое рисование без процессной логики;
- любой редактор схем сам по себе;
- любая таблица Excel без процессной модели;
- AI-чат без описания шагов, ролей и результата.

## 3. Чем отличаются BPM, BPMN и ARIS

| Понятие | Тип | Простое объяснение | Пример |
|---|---|---|---|
| BPM | направление | управление процессами | улучшить подготовку билетов |
| BPMN | нотация | язык схем | нарисовать процесс подготовки билета |
| ARIS | платформа / методология | среда моделирования | описать процессы, роли, документы |
| Camunda | workflow engine | запуск процессов | автоматизировать согласование |
| Draw.io | редактор диаграмм | нарисовать схему | сделать простую BPMN-схему |

### Главное отличие одной фразой

- BPM отвечает: **зачем управлять процессом**;
- BPMN отвечает: **как описать процесс единым языком**;
- ARIS отвечает: **в какой среде связать процесс с архитектурой организации**.

## 4. Основные элементы BPMN

### Событие

- **Определение:** то, что случается: старт, сообщение, ошибка, завершение.
- **Пример:** выбран экзаменационный билет; лекция создана; OCR не распознал текст.
- **В личной системе:** фиксировать точку входа и точку готовности в Exam OS и Research OS.

### Задача

- **Определение:** конкретное действие, которое нужно выполнить.
- **Пример:** извлечь тезисы; создать карточку билета; сделать устную версию ответа.
- **В личной системе:** превращать шаги учёбы и проектов в понятные действия для Cursor.

### Шлюз

- **Определение:** точка решения или ветвления процесса.
- **Пример:** источников достаточно? нужен ли OCR? билет готов к устному ответу?
- **В личной системе:** показывать развилки качества и не идти дальше при пробелах.

### Поток

- **Определение:** порядок шагов: что за чем следует.
- **Пример:** тезисы → лекция → полный ответ → устная версия.
- **В личной системе:** строить учебные pipelines без хаоса.

### Пул

- **Определение:** граница участника процесса: человек, роль или система.
- **Пример:** пул «Студент», пул «Cursor», пул «Google Calendar».
- **В личной системе:** разделять, что делает человек, а что делает инструмент.

### Дорожка

- **Определение:** зона ответственности внутри пула.
- **Пример:** в пуле «Студент» — чтение, конспект, устная тренировка.
- **В личной системе:** уточнять, какая часть работы относится к какой роли или режиму.

### Сообщение

- **Определение:** обмен данными между участниками процесса.
- **Пример:** API передаёт событие в календарь; система отправляет уведомление.
- **В личной системе:** описывать интеграции Excel → Web → API → Calendar.

### Документ

- **Определение:** артефакт, который процесс читает или создаёт.
- **Пример:** PDF, markdown-лекция, карточка билета, чеклист готовности.
- **В личной системе:** связывать шаги процесса с конкретными файлами экосистемы.

### Подпроцесс

- **Определение:** крупный шаг, внутри которого есть свой мини-процесс.
- **Пример:** «Анализ книги» внутри «Подготовки билета».
- **В личной системе:** не перегружать главную схему деталями Research OS или Life OS.

## 5. Типовые инструменты

| Инструмент | Роль в категории | Когда использовать |
|---|---|---|
| ARIS | комплексное моделирование организации | процессы + роли + данные + архитектура |
| Camunda | исполнение процессов | когда процесс нужно запускать |
| Bizagi | BPM suite | моделирование и автоматизация вместе |
| Signavio | process intelligence / BPM | анализ и управление процессами |
| Draw.io | редактор диаграмм | быстрая учебная BPMN-схема |
| Miro | visual collaboration | совместное проектирование процесса |
| Visio | diagramming / modeling | корпоративные схемы и документация |
| Lucidchart | online diagramming | совместные диаграммы в облаке |

### Правило выбора

1. Нужна глубина архитектуры — ARIS.
2. Нужна быстрая схема для себя — Draw.io или markdown.
3. Нужно вместе подумать — Miro.
4. Нужно исполнять процесс — Camunda / Bizagi.

## 6. Какие данные использует BPM-система

- процессы;
- роли;
- документы;
- события;
- задачи;
- решения;
- статусы;
- правила;
- показатели;
- входы и выходы.

### Как читать эти данные

| Тип данных | Вопрос |
|---|---|
| Процессы | Что повторяется? |
| Роли | Кто отвечает? |
| Документы | Что читаем и создаём? |
| События | Что запускает и завершает работу? |
| Задачи | Какие действия обязательны? |
| Решения | Где развилки и проверки? |
| Статусы | На каком этапе сейчас процесс? |
| Правила | Что нельзя нарушать? |
| Показатели | Как понять, что процесс улучшился? |
| Входы и выходы | Что входит и что получается? |

## 7. Какие проекты можно создать в Cursor

### 1. BPMN-карта подготовки к экзамену

- **Что сделать:** описать путь от списка билетов до готовности отвечать.
- **Зачем:** увидеть Exam OS как единый процесс.
- **Результат:** карта экзаменационной подготовки.
- **Польза:** понятны узкие места и порядок работы.

### 2. BPMN-карта анализа книги

- **Что сделать:** книга → главы → тезисы → понятия → сравнение.
- **Зачем:** стандартизировать Research OS.
- **Результат:** повторяемый исследовательский процесс.
- **Польза:** меньше хаоса при работе с литературой.

### 3. BPMN-карта создания лекции

- **Что сделать:** тема → источники → структура → полный текст → конспект → вопросы.
- **Зачем:** сделать создание лекций управляемым.
- **Результат:** процессный стандарт лекции.
- **Польза:** Cursor получает точный сценарий генерации.

### 4. BPMN-карта Life OS

- **Что сделать:** сферы жизни → обзор → малые шаги → контроль нагрузки.
- **Зачем:** стабилизировать жизнь рядом с учёбой.
- **Результат:** процесс еженедельной устойчивости.
- **Польза:** снижение перегруза и паники.

### 5. BPMN-карта обработки PDF

- **Что сделать:** PDF → извлечение → проверка → markdown → карточка.
- **Зачем:** ускорить работу с источниками.
- **Результат:** document pipeline.
- **Польза:** ясно, где нужен OCR и контроль качества.

### 6. BPMN-карта Excel → Web → API

- **Что сделать:** описать цепочку данных от таблицы до внешней системы.
- **Зачем:** проектировать интеграции без хаоса.
- **Результат:** архитектура data flow.
- **Польза:** основа для будущего web/API-проекта.

### 7. BPMN-карта Google Calendar

- **Что сделать:** событие → проверка → создание → уведомление → обзор.
- **Зачем:** связать время и автоматизацию.
- **Результат:** календарный workflow.
- **Польза:** понятный контур личной продуктивности.

### 8. BPMN-карта портфолио

- **Что сделать:** достижение → критерий комиссии → карточка → размещение.
- **Зачем:** готовить портфолио системно.
- **Результат:** процесс Portfolio OS.
- **Польза:** каждый элемент усиливает нужный критерий.

### 9. BPMN-карта повторения билетов

- **Что сделать:** выбор билета → краткая версия → устная тренировка → ошибки → повтор.
- **Зачем:** сделать повторение регулярным.
- **Результат:** цикл закрепления.
- **Польза:** рост готовности к комиссии.

### 10. BPMN-карта создания HTML-доски

- **Что сделать:** содержание → структура → HTML → проверка в браузере → правки.
- **Зачем:** не начинать web-панель слишком рано.
- **Результат:** безопасный процесс создания интерфейса.
- **Польза:** соблюдается правило «сначала содержание, потом код».

## 8. Как применять в моей системе

### Exam OS

Описывать подготовку билета, повторение, самопроверку и путь к устному ответу.

### Research OS

Описывать разбор книги, извлечение тезисов, сравнение источников и синтез.

### Knowledge OS

Показывать, как знание проходит путь: источник → тезис → термин → карточка → повторное использование.

### Life OS

Описывать обзор недели, малые шаги, контроль нагрузки и антипаника-сценарии.

### Project OS

Перед мини-проектом сначала рисовать процесс: идея → категория → данные → шаги → результат.

### Visualization OS

Хранить BPMN-схемы, карты процессов и визуальные модели в `12_Visualization_OS_визуализация`.

### Практическое правило для Cursor

```text
Сначала BPMN-процесс
→ потом данные и файлы
→ потом структура проекта
→ потом автоматизация / HTML / API
```
